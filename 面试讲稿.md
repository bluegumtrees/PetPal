## 一、项目名片（30 秒讲完版）

**一句话简介**：
> 一个**真正的多模态 AI Agent**——基于 **VLM + OpenAI Function Calling** 把"拍照 + 文字"输入路由到 5 类任务（症状识别 / 体态评分 / 疼痛评估 / 情绪解读 / 闲聊），主控 LLM 自主编排 6 个 tool（兽医 RAG / 高德地图 / 时间线读写 / 重看图 / 计划提醒）完成端到端服务，**临床疼痛量表 FGS（Evangelista 2019）prompt 化**为 VLM 输出 schema，前端用 SSE 卡片流可视化每一步 agent 决策，**257 条中文兽医 KB + APScheduler 提醒 + 时序对比图 + P7 评测 96.7% pass**。

**11 大亮点**：

1. **真 Agent 不是 chatbot**：LLM 自主决策调哪些 tool、按什么顺序，全程审计 `tool_calls_json` 可追溯（前端 ToolCallCard 折叠展开 args/result）
2. **临床量表 prompt 化 ⭐⭐⭐**：把同行评议的 Feline Grimace Scale（5 AU × 0/1/2 评分）编码为 gpt-4o-mini 的 VLM schema，让通用 VLM 输出医学级结构化评估，**项目最强差异化**
3. **多 task 路由 + 上下文继承**：用便宜文本模型先分类到 5 task（symptom/bcs/pain_fgs/emotion/chat），比让 VLM 路由便宜 5-10×；多轮对话中无文字+图能从上一轮 assistant content 继承 task
4. **257 条 KB + 三阶段检索 ⭐**：129 条手写骨架（参考 PetMD/Cornell）+ 128 条 **MSD Veterinary Manual 翻译**（Merck Manual 兽医百科金标准），GPT-4o 辅助翻译 + 人工 review，急诊红线 verbatim 保留；dense(BGE) + sparse(BM25+jieba) → RRF → CrossEncoder rerank
5. **时序对比图 + 体重四入口单一来源 ⭐**（P6.1）：BCS/体重/FGS 时序图（Recharts），三处 task 数据源解耦——BCS/FGS 直接读 chat_sessions.vlm_output_json（不依赖 LLM 调 save_event 命中率），体重读 pet_events；档案/编辑/inline 称重/LLM 口述四条入口经 `update_pet` 钩子统一到唯一 ground truth
6. **APScheduler 提醒 + 单一 source of truth ⭐**（P6.2）：AsyncIOScheduler timezone=UTC、内存 jobstore（**拒绝 SQLAlchemyJobStore 避免双 source of truth**）、启动恢复 1h 阈值（过期 <1h 补发 / >1h stale）、dry-run 模式（未配 SMTP 时存完整邮件内容到 db，演示用）
7. **历史 VLM 进 context 而不重跑 VLM ⭐**（P6.3）：本轮无图时从 chat_sessions 拿上次 VLM 输出注入 context（含 rationale），LLM 可基于历史分析直接回答；想看新细节调 reanalyze_image with focus（智能兜底：同 task + 空 focus 拒绝，换 task 允许）
8. **SSE 流式 + 卡片流 UI**：async generator + asyncio.to_thread 把同步 LLM/RAG/VLM 包装为流式事件，前端 fetch + ReadableStream 解析（EventSource 不支持 POST multipart），每步 yield 不同事件
9. **多 entity 数据模型 + 软删**：4 表 SQLModel（pets / pet_events / reminders / chat_sessions），支持多只宠物切换 + 软删保留 + 头像 Pillow resize + chat_sessions 含 `session_id / is_intermediate / vlm_output_json / tool_calls_json` 完整审计
10. **P7 评测 6 轮迭代 ⭐⭐**：retriever 32 query × 4 配置（hit@5 96.7%，rerank 单独 +27% hit@1）+ agent E2E 10 case × 3 runs 迭代 40%→53%→80%→83%→**96.7%**（修 sync/stream 双轨漂移 + prompt phase 概念 + 历史 VLM context + reanalyze 工程兜底）；OOK robustness top-1 score 差距 0.6+ 可作 agent 判断"无相关知识"信号
11. **Prompt 工程边际收益递减实测 + 工程兜底 ⭐**：从死规则迭代到引导式 + 多个工程兜底（`_looks_like_transition_only` retry / `saved_event_keys` 去重 / 体重 note 兜底提取 X kg / 历史 VLM 进 context / reanalyze 智能拒绝）；评测发现 a02 / a03 "明确事件型 query 不查 RAG" 是 prompt 引导无法触达的 corner case → **主动选择不堆规则**

**技术栈**：Python 3.12 / FastAPI / SQLModel + SQLite / ChromaDB / sentence-transformers (BGE-small-zh + BGE-reranker-base) / rank-bm25 + jieba / OpenAI SDK + OpenRouter (gpt-4o-mini / gpt-4o for KB translation) / Pillow (VLM 图片预处理) / 高德 Web API / **APScheduler (P6.2)** / BeautifulSoup4 + lxml (MSD 抓取) / **前端**：React 19 + Vite 8 + Tailwind v4 + react-router-dom 7 + **Recharts (时序图)** + react-markdown + JSDoc + 原生 fetch ReadableStream 流式

## 二、简历描述（粘贴版）

**PetPal 智能宠物管家** — 个人项目 · 2026.0X – 2026.0X

**项目简介**：多模态 AI Agent，对宠物主人「拍照 + 文字」输入做症状识别 / 体态评分 / 疼痛评估 / 情绪解读 / 计划提醒，基于 **VLM + OpenAI Function Calling** 自主编排 6 个工具（兽医 RAG / 高德地图 / 事件时间线 / 重看图 / schedule_reminder）完成端到端服务，临床疼痛量表 FGS prompt 化为 VLM 输出 schema，**257 条中文兽医 KB + APScheduler 提醒 + 时序对比图 + P7 评测 hit@5 96.7% / agent 96.7% pass**。

**技术栈**：Python 3.12、FastAPI、SQLModel + SQLite、ChromaDB、sentence-transformers (BGE-small-zh + BGE-reranker-base)、rank-bm25 + jieba、OpenAI SDK + OpenRouter (gpt-4o-mini / gpt-4o)、APScheduler、BeautifulSoup4 (MSD 抓取)、React 19 + Vite 8 + Tailwind v4 + react-router-dom 7 + **Recharts**、原生 fetch ReadableStream SSE 流式

**核心负责**：

- 设计**多模态 Agent 主循环**（`async generator` + `asyncio.to_thread`）：VLM 首步 → 5 task 路由 → 6 个 Function Calling tool 编排 → SSE 事件流输出，单次会话最多 5 轮 iter，完整记录 `tool_calls_json` 决策审计可追溯。
- 设计**临床疼痛量表 prompt 化**：把同行评议 FGS（Evangelista 2019, Scientific Reports）的 5 个 Action Units × 0/1/2 评分编码为 gpt-4o-mini 的结构化输出 schema，让通用 VLM 输出医学级评估（耳朵位置 / 眼睑紧张 / 口鼻紧张 / 胡须变化 / 头部位置 各维度评分 + total/10 + normalized 阈值判断）。
- 实现**三阶段检索 RAG + 知识库 P1.5 翻倍**：手写 129 条中文骨架 + **MSD Veterinary Manual（Merck 默克兽医百科金标准）翻译 128 条**（GPT-4o 辅助 + 人工 review，急诊红线 verbatim 保留），共 257 条 YAML frontmatter 知识库；dense(BGE-small-zh + Chroma) + sparse(BM25 + jieba) → RRF 融合 → CrossEncoder rerank。
- 实现**多 entity 数据模型**（SQLModel + SQLite，4 表）：pets（含软删 deleted_at）、pet_events（时间序列）、reminders（含 delayed_reason / notification_payload_json）、chat_sessions（含 session_id / is_intermediate / vlm_output_json 完整审计），头像 Pillow resize + 时间戳防缓存。
- 实现 **P6.1 时序对比图 + 体重四入口单一来源**：Recharts 时序图（BCS / 体重 / FGS 三 tab，窗口切换 7/30/全部），数据源解耦——BCS/FGS 直接读 chat_sessions.vlm_output_json 不依赖 LLM 调 save_event 命中率，体重读 pet_events；档案编辑 / inline 称重 / LLM 口述 / 创建初始 四条入口经 `update_pet` 钩子统一同步 pets.weight_kg 与 pet_events.weight 双向一致。
- 实现 **P6.2 APScheduler 提醒系统**：AsyncIOScheduler timezone=UTC、**内存 jobstore（拒绝 SQLAlchemyJobStore 避免双 source of truth）**、启动恢复策略（过期 <1h 补发 + delayed_reason、过期 >1h stale 不发避免连发）、dry-run 邮件模式（未配 SMTP 时存完整邮件内容到 db 演示用）、LLM 入口 `schedule_reminder` tool（本地时间 → UTC 容错多种格式 + 过去时间拒绝）、dev `trigger_now` 立即触发（PETPAL_DEV_MODE 守护）。
- 实现**工程兜底集**：① `_looks_like_transition_only` 检测 LLM "光说不做" + system 提示重试（sync/stream 双轨同步）；② `saved_event_keys` 去重 save_pet_event；③ 体重 note 兜底（LLM 把 X kg 塞 note 没填 payload → 正则提取后同步档案）；④ **历史 VLM 进 context**（无图轮从 chat_sessions 拿上次 VLM 注入，LLM 看得到 rationale 无需 reanalyze）；⑤ `_reanalyze_image` 智能拒绝（同 task + 空 focus 拒绝，换 task 允许）。
- 实现 **P7 完整评测体系**：retriever 32 query × 4 配置对比（dense/sparse/hybrid/hybrid+rerank，hit@5 96.7%，rerank 单独 +27% hit@1，OOK robustness 验证）+ agent E2E 10 case × 3 runs 多轮迭代（40%→53%→80%→83%→**96.7%**）；**完整评测 → 诊断 → 修复 → 验证闭环**，每轮都暴露具体工程问题（sync/stream 漂移 / prompt phase 概念 / 评测口径校准 / 历史 VLM context 注入）。
- 实现 **React 多页 chat UI**：React 19 + Vite + Tailwind v4 + react-router-dom 7 路由（/chat、/dashboard、/pets/:id、/dev/vet-search）、卡片流（UserMessage / TaskBadge / VLMCard / ToolCallCard / AssistantMessage / AssistantThinking）、时序图 + 称重 inline 输入 + reminders section + AddEventModal、per-pet session localStorage 持久化、duplicate/skipped tool 卡片自动隐藏、Lightbox 全屏图片。
- 实现 **SSE 流式后端**：FastAPI StreamingResponse 暴露 12+ 种 SSE event 类型，前端 fetch + ReadableStream 手动解析（EventSource 不支持 multipart 上传）。
- **Prompt 工程边际收益递减实测**：从死规则迭代到引导式 + 工作基调 phase 概念（区分 motivation 短句 vs final 完整答复）；评测主动暴露 prompt 引导的天花板（明确事件型 query LLM 不查 RAG），主动选择**不堆规则**避免输出模板化（P5 已踩坑）。

## 三、业务背景与需求

### 1. 为什么做这个

- **痛点**：中国宠物饲养量近 1.3 亿，但宠物医疗就诊门槛高（费用 / 排队 / 距离）。主人有大量「小问题想问问」的需求——「吐了 2 次要不要看医生」、「这个体型算不算胖」、「为什么躲着我」——传统兽医客服解决不了
- **传统方案不足**：① 兽医论坛响应慢 + 信息真假难辨；② 万能客服 AI 没有专业兽医知识，回答容易翻车；③ 看图识病的 App 通常只做单一功能（比如只识别皮肤病），不能跨任务
- **AI 切入点**：**VLM + Function Calling + RAG 三件套**能形成「拍照 + 查知识库 + 调地图 + 持久化档案」的完整闭环，让一个 agent 能覆盖多场景——这正是 AI 应用岗最值得做的事

### 2. 业务场景

| task | 用户行为 | 系统响应 |
|---|---|---|
| chat | 「我家猫可爱吗？」（带图） | VLM 看图给 observation，agent 自然温和回应，**不医学化** |
| symptom | 「我家猫吐了 2 次没精神」 | RAG 检索 + 综合建议 + save_event 入库 + 急诊时给就医标准 |
| bcs | 「看看它体态」（带图） | VLM 给 BCS 1-9 分 + 评分依据，极端值时查 RAG 拿专业饮食方案 |
| pain_fgs | 「它不舒服是不是疼？」（带图） | VLM 给 FGS 5 AU 评分 + total_score/10 + normalized 阈值判断 |
| emotion | 「它最近躲着我」 | VLM 给 body_signals + candidate_emotions + confidence，给互动建议 |

**6 个 Function Calling Tools 主控编排**：
1. `retrieve_vet_knowledge` — 兽医知识库 RAG
2. `find_nearby_clinic` — 高德地图查附近宠物医院
3. `save_pet_event` — 写时间线（每次关键发现入库）
4. `query_pet_history` — 查历史事件
5. `reanalyze_image` — 多轮重看图（换 task 或换 focus）
6. ~~`send_alert_email`~~ — P5 阶段移除（主人就在 chat 前看，不需要邮件给自己；P6 会重设计为 `schedule_reminder` 用于日程提醒）

### 3. 为什么我会想到这个

- **个人触发**：身边朋友养猫养狗，常问"它这个状态正常吗 / 该不该看医生"，没有可靠 AI 工具能给具体建议
- **技术契合**：多模态 + Function Calling 是 2026 年 AI 应用岗的高频技能，但应聘者大多只做过单功能 chatbot，**端到端多模态 agent** 项目少
- **数据可控**：中文兽医专科数据集稀缺（HF 上几乎没像样的），自己手写 130 条 + 翻译 PetMD 兜底——**质量 > 数量**，面试讲"我整理了 130 条同行评议来源的种子库"比"我爬了 10000 条来源不明的"可信得多
- **临床量表突破口**：搜文献时找到 Evangelista 2019 的 FGS 论文（Scientific Reports 同行评议），意识到可以**把临床量表 prompt 化为 VLM schema**——这是项目独有的创新点

## 五、核心技术决策

### 决策 1：真正的 Agent，不是 Chatbot ⭐

**问题**：LLM 应用市面上 90% 是 chatbot——固定流水线（检索 → prompt → 生成），LLM 没有决策权。

**解决**：用 OpenAI 真 Function Calling 协议，让 LLM 自主决定调哪些 tool、按什么顺序：

```python
async def run_agent_stream(...):
    for iteration in range(max_iter):
        resp = await client.chat.completions.create(
            messages=messages, tools=TOOL_SCHEMAS, ...
        )
        if not resp.tool_calls:
            yield final_answer
            return
        for tc in resp.tool_calls:
            result = TOOL_DISPATCH[tc.name](json.loads(tc.arguments), ctx)
            yield tool_result
            messages.append({"role": "tool", ...})
```

**讲法**：
> "我做的是真 agent——LLM 看到用户消息后自己决定调 retrieve_vet_knowledge 还是 find_nearby_clinic、调几次、按什么顺序。整个 `tool_calls_json` 持久化到 chat_sessions 表，**决策可追溯**——这是 agent 和 chatbot 的本质区别。"

### 决策 2：临床量表 FGS Prompt 化 ⭐⭐⭐

**问题**：让 VLM 评估宠物疼痛——通用 prompt 会输出主观描述（"看起来不太舒服"），不够医学严谨；自训分类器需要数据集和算力。

**解决**：把 **Evangelista 等 2019 年发表在 Scientific Reports 的 Feline Grimace Scale**（FGS）的 5 个 Action Units 评分标准，**编码为 gpt-4o-mini 的 VLM 输出 schema**：

```json
{
  "action_units": [
    {"name": "Ear position", "score": 0-2, "rationale": "..."},
    {"name": "Orbital tightening", "score": 0-2, "rationale": "..."},
    {"name": "Muzzle tension", "score": 0-2, "rationale": "..."},
    {"name": "Whiskers change", "score": 0-2, "rationale": "..."},
    {"name": "Head position", "score": 0-2, "rationale": "..."}
  ],
  "total_score": 0-10,
  "normalized": 0.0-1.0,
  "interpretation": "解释总分相对 0.39 临床阈值",
  "recommendation": "...",
  "source": "Evangelista et al. 2019, Scientific Reports"
}
```

每个 AU 在 prompt 里给完整的 0/1/2 视觉判定标准（论文训练手册原文），让通用 VLM 「对号入座」。

**讲法**：
> "面试官如果问我'怎么让 LLM 做医学评估'——我答：**把同行评议的科学量表编码为 VLM 的 prompt schema**。FGS 这种 5×3 评分矩阵是视觉描述（不需要音频/时序），VLM 对号入座准确率高。我没有训分类器，但通过 prompt 工程让通用模型输出医学级结果——**这是 LLM 时代的范式优势**。"

### 决策 3：cheap 文本路由器，而不是 VLM 决定 task

**问题**：用户输入「拍照 + 文字」，怎么决定走哪个 task？让 VLM 看图后决定？

**解决**：用一次 cheap gpt-4o-mini **文本调用**做 5 类分类（chat/symptom/emotion/bcs/pain_fgs），比让 VLM 路由便宜 5-10×。规则：

- 无文字 + 无图 → 报错
- 无文字 + 有图 → 默认 chat（兜底）+ 查历史 assistant content 关键词继承 task
- 有文字 → cheap 模型路由

**讲法**：
> "我评估过两种路由：① VLM 看图后决定 task；② cheap 模型看文字决定 task。前者 token 成本高 5-10 倍，且如果 VLM 路由错了还要重新跑 VLM 一次更贵。**前任 Claude 的判断**：『VLM 调用比 router 贵 5-10 倍，省路由反而可能多跑一次 VLM』——这种工程权衡很值钱。"

### 决策 4：多 task VLM Prompt 设计（4 task × Pydantic schema）

**问题**：不同任务（症状 / 情绪 / BCS / 疼痛）VLM 输出结构完全不同，怎么统一？

**解决**：每个 task 一个独立 prompt + Pydantic schema：

| Task | Prompt 角色 | 输出 schema |
|---|---|---|
| symptom | 临床兽医 | `observation / possible_symptoms[] / severity / urgency / recommendation` |
| emotion | Feline/Canine Behaviorist | `body_signals[] / candidate_emotions[] / caveat / suggestion` |
| bcs | 宠物营养师（WSAVA 9 分制） | `bcs_score 1-9 / rationale / diet_advice / caveat` |
| pain_fgs | Feline Behaviorist + 兽医 | `action_units[5] / total_score / normalized / interpretation` |

调用统一接口 `analyze(image_path, task, species, extra)`，内部 Pydantic 校验 + 失败重试 1 次（零温度 + 错误提示）。

**讲法**：
> "我特意让 emotion 任务**不断言情绪**，输出 `candidate_emotions + confidence`——因为单图无法判断动态信号（尾巴抽打频率、呼噜），断言会误导。这是**诚实工程权衡**——告诉面试官我知道 VLM 单图的局限，用 schema 设计兜住。"

### 决策 5：三阶段检索复用（同 ElderCare 工业标配）

**问题**：兽医知识库怎么检索？

**解决**：复用 ElderCare 思路——dense + sparse → RRF → rerank。但**优化点**：

- 知识库自写 130 条 + YAML frontmatter（不爬国内站，版权/反爬/伦理三重风险）
- 每条 8 字段 metadata：`species / severity / age_group / emergency / tags / source / source_url`
- 支持 `where: {species_cat: true, emergency: true}` 等 metadata filter
- chunk = 一个二级标题块（约 200-400 字，中文 RAG 最佳长度）

**讲法**：
> "ElderCare 的检索三件套我直接复用过来——dense（BGE-small-zh + Chroma）+ sparse（BM25 + jieba）→ RRF 融合 → CrossEncoder rerank。PetPal 的优化在于 **metadata 设计**：YAML frontmatter 让每条带 emergency/severity 字段，前端能筛选「只看急诊」，**这是 ElderCare 没做的**。"

### 决策 6：知识库自写 + PetMD 翻译兜底（不爬国内站）

**问题**：中文兽医专科数据集 HF 上几乎没有，怎么获取知识库？

**解决**：**自己手写 130 条**（11 个主题文件，按系统分类）+ 翻译 PetMD 英文权威站点扩长尾（计划 P1.5 阶段）。**不爬国内站**——反爬 + 版权 + 伦理三重风险，作为学生项目得不偿失。

**讲法**：
> "中文兽医专科数据极少，我评估过三条路径：① 爬国内站（反爬 + 版权风险）；② 翻译 AKC/PetMD 英文权威；③ 自写种子库。最终选 **2+3 混合**——自写 130 条核心 + PetMD 翻译扩长尾。**面试讲『我整理了 130 条同行评议来源的种子库』** 比 『我爬了 10000 条来源不明的』可信得多。"

### 决策 7：DB 4 表 + SQLModel + 软删

**问题**：多只宠物 + 时间序列事件 + chat 历史，怎么设计 schema？

**解决**：4 表 SQLModel + 软删：

```python
pets             # 档案 + photo_url + deleted_at（软删）
pet_events       # 时间序列「已发生」（BCS / 症状 / 疫苗 / pain_fgs / emotion）
reminders        # 「未发生」计划（P6 才真用）
chat_sessions    # 对话历史，P5 加 session_id / is_intermediate / vlm_output_json / tool_calls_json
```

**关键**：events 和 reminders **分开**——events 不可变历史，reminders 可改可消。

**讲法**：
> "events / reminders 分开是产品决策——events 是『已发生』不可变，reminders 是『未发生』可改可消。混在一起状态机乱。**`chat_sessions.tool_calls_json` 是面试讲点**——agent 每一步调了什么 tool 都能审计，前端 ToolCallCard 能展开看 args 和 result。"

### 决策 8：Context Engineering > Tool Engineering

**问题**：agent 每次都调 query_pet_history 拿宠物档案太浪费 token。

**解决**：每次 agent 启动**自动注入** system context：

```
当前宠物档案：亚历山大（猫，奶牛猫，1岁2月，公，已绝育，4.5kg，pet_id=3）
最近 5 条事件：
  - 2026-05-10 vaccine: 猫三联
  - 2026-04-15 bcs: 5/9
  - ...
```

LLM 看到上下文就知道，不必再调工具。

**讲法**：
> "**Context Engineering > Tool Engineering**——把最常用的信息（宠物档案 + 最近事件）直接注入 system context，比让 agent 反复调 query_pet_history 便宜 + 快。这条优化让 LLM 调用从平均 3-4 轮降到 2-3 轮。"

### 决策 9：「光说不做」检测 + System 注入兜底 ⭐

**问题**：gpt-4o-mini 偶尔输出「我先查一下兽医知识库」但 `tool_calls` 为空——光说不做。Agent loop 看到 no tool_calls 就当 final 终止，用户体验糟糕。

**解决**：工程兜底——检测 LLM 输出是否为「过渡语模式」（含动作词 + 内容短 + 无具体阈值数字），如果是 → 持久化为 thinking + 注入 system 提示让 LLM 重试一轮：

```python
def _looks_like_transition_only(text):
    if len(text) > 200: return False
    if not any(w in text for w in _TRANSITION_HINTS): return False
    has_specifics = any(d in t for d in ['24 小时', '禁食', '## ', '- ', '1.'])
    return not has_specifics

# agent loop
if not msg.tool_calls and _looks_like_transition_only(final):
    messages.append({'role': 'system', 'content': '请立即调对应 tool 真正执行——只说不做等于放弃任务。'})
    continue
```

**讲法**：
> "这是 LLM 应用工程化的典型坑——**模型行为非确定性，prompt 引导只能 80% 命中**。我设计了「光说不做」检测兜底——文本特征匹配 + system 注入重试，把 LLM 漏行为兜回来。**核心思想**：不依赖 LLM 完全听话，关键业务规则代码强制。"

### 决策 10：自动补存 + 历史图自动回填 ⭐

**问题 1**：LLM 给 final 后忘记调 save_pet_event（事件没入时间线）
**问题 2**：多轮对话用户上传一次图后，后续讨论时 LLM 没调 reanalyze_image，agent 让用户重发图（体验差）

**解决**：两个工程兜底：

```python
# 兜底 1：final 阶段自动补存
needs_save = task in ('symptom', 'bcs', 'pain_fgs', 'emotion')
if needs_save and (pet_id, task) not in saved_event_keys:
    auto_payload = _build_auto_payload(task, vlm_output, user_text)
    silent_save_pet_event(auto_payload)
    # 不 yield SSE 事件 → UI 静默；db 有 audit

# 兜底 2：本轮无图 + 有历史图 → 把历史图当本轮图
if not has_image and historical_image_path and task in ('bcs', 'pain_fgs', 'emotion', 'chat'):
    image_path_str = historical_image_path
    has_image = True
    # VLM 自动跑，LLM 收到的状态 = "本轮有图"
```

**讲法**：
> "Prompt 引导 LLM 调 reanalyze_image 在多轮对话里不可靠（漏调率 50%+）。我用**工程兜底彻底解决**——后端 planner 检测本轮无图但 session 有过图时，直接把历史磁盘路径当本轮 image_path 注入。**完全不依赖 LLM 决策**——这是 LLM 不可靠时的正解。"

### 决策 11：Prompt 工程 8 版本演进——从死规则到引导式 ⭐

**问题**：每次加 ❌ 绝对禁止规则，LLM 输出反而越来越机械。

**演进路径**（讲面试时重点讲这条）：

| 版本 | 思路 | 问题 |
|---|---|---|
| v1-v3 | 堆 SOP / 工具流程 / 禁止条目 | LLM 像填表机器，无温度 |
| v4 | 强化 thinking 引导（具体例子） | LLM 把整段建议塞 thinking，final 反而空洞 |
| v5 | 严限 thinking ≤30 字 + 强 final | LLM 干脆什么都不写 |
| v6 | 「教练式」引导，删冗余禁令 | 仍非确定 |
| v7 | 完整重构 + 历史图工程兜底 | 较稳但 LLM 偶尔不存事件 |
| **v8** | **简化为引导式 + 关键规则用工程兜底** | **目前最稳，约 2000 字** |

**讲法**：
> "我的 prompt 改了 8 版。最大教训：**LLM 不是程序员，prompt 不是代码**。堆规则只会让模型机械，应该给**人设 + 例子 + 必要约束 + 工程兜底关键行为**。从 3300 字砍到 2000 字，效果反而更好——这是 LLM 应用工程化的精髓。"

### 决策 12：SSE 流式 + Async Generator + asyncio.to_thread

**问题**：怎么让前端能可视化 agent 的每一步决策？

**解决**：planner 写成 `async def run_agent_stream() -> AsyncIterator[dict]`，yield 12+ 种事件：

```python
async def run_agent_stream(...) -> AsyncIterator[dict]:
    yield {'type': 'start', 'session_id': ...}
    task = await asyncio.to_thread(classify_task, ...)
    yield {'type': 'task_classified', 'task': task}
    if has_image:
        yield {'type': 'vlm_start', 'task': vlm_task}
        vlm_output = await asyncio.to_thread(vlm_analyze, ...)
        yield {'type': 'vlm_done', 'output': vlm_output}
    for iteration in range(max_iter):
        yield {'type': 'iter_start', 'iter': iter+1}
        resp = await asyncio.to_thread(client.chat.completions.create, ...)
        for tc in resp.tool_calls:
            yield {'type': 'tool_call', 'tool': tc.name, 'args': args}
            result = await asyncio.to_thread(TOOL_DISPATCH[tc.name], args, ctx)
            yield {'type': 'tool_result', 'tool': tc.name, 'result': result}
        ...
    yield {'type': 'final_answer', 'content': final}
    yield {'type': 'done', 'iterations': ..., 'elapsed_s': ...}
```

FastAPI `StreamingResponse` 把 yield 的事件序列化为 `data: {...}\n\n` SSE 帧。

**讲法**：
> "同步 LLM/RAG/VLM 调用用 `asyncio.to_thread` 扔线程池，避免阻塞 event loop。复用 ElderCare 的 `event_gen()` 模式，但事件类型更丰富（chat agent 比单纯 RAG 多了 task/vlm/tool 这些维度）。**这是 ReAct 风格 agent 的标准实现**。"

### 决策 13：前端卡片流 + UIBlock 设计

**问题**：怎么把 12 种 SSE 事件渲染成清晰的对话流？

**解决**：前端用 `UIBlock[]` 数组，每个 SSE 事件 → 创建/更新一个 block：

```javascript
const blocks = [
  { kind: 'user', data: { content, imageUrl, vlmTask, vlmOutput } },
  { kind: 'task', data: { task } },                    // 🏷 task badge
  { kind: 'vlm', data: { task, output } },             // 📷 VLMCard
  { kind: 'thinking', data: { content } },             // 💭 AssistantThinking（≤40字短显示 / >40字折叠）
  { kind: 'tool', data: { tool, args, status, result, summary } },  // 🔧 ToolCallCard（running→done）
  { kind: 'assistant', data: { content } },            // 🐾 AssistantMessage（react-markdown）
]
```

duplicate/skipped tool result 自动从数组移除（UI 不显示，db 有 audit）。

**讲法**：
> "UIBlock 设计灵感来自 ChatGPT 的对话流——每个事件一张卡，时间顺序明确。tool 卡片 running 状态有 🟠 闪烁，done 后 ✓，error 后 ✕，点击展开看完整 args/result。**这是 agent 体验和普通 chatbot 的最大视觉差异**——主人能看到 agent 在做什么。"

### 决策 14：多页路由 + per-pet long-running session

**问题**：用户切换不同宠物时，会话怎么管理？

**解决**：
- React Router v7 多页：`/`（chat）、`/dashboard`、`/pets/:id`、`/dev/vet-search`
- `useSession(petId)` hook：localStorage 存 `{petpal.sessions: {pet_id: session_id}}` 映射
- 切换宠物 → 自动切换对应 session_id
- 「+ 新对话」按钮：当前 pet 生成新 UUID 覆盖

**讲法**：
> "前任 Claude 的判断很对：**关 tab 就丢历史会崩用户**——宠物场景跨天连续是核心体验。我用 localStorage 持久化 per-pet session_id，刷新页面自动恢复，切宠物自动切对应历史。LLM 上下文取最近 8 轮，避免 token 爆炸。"

## 六、前置知识速查

### 1. VLM（Vision Language Model）原理

- **VLM = Vision Encoder + LLM**：图片先过视觉编码器（如 CLIP / ViT）转向量，再喂给 LLM 当上下文
- **gpt-4o-mini vision**：OpenAI 的多模态轻量版，支持 `image_url` 类型的 message content，base64 或公网 URL
- **本项目**：图片 Pillow resize 到 1024×1024 + JPEG 85 质量 + base64 编码，单图成本 ~$0.001-0.002

### 2. OpenAI Function Calling 协议

```python
tools = [{
    "type": "function",
    "function": {
        "name": "retrieve_vet_knowledge",
        "description": "...",
        "parameters": {"type": "object", "properties": {...}, "required": [...]}
    }
}]
resp = client.chat.completions.create(model=..., messages=..., tools=tools)
# resp.choices[0].message.tool_calls → 含 [tool_call_id, function.name, function.arguments]
```

**关键**：tool_call 是 LLM 决定的，**不是 prompt 工程触发**——这才是真 agent。

### 3. ReAct Agent Loop

```
循环（max_iter 次）:
  1. LLM(messages + tools) → response
  2. if response.tool_calls:
       for each tc: execute(tc) → tool_result; messages.append(...)
  3. else:
       return response.content as final answer
```

本项目用 max_iter=5，绝大多数 case 2-3 轮内收敛。

### 4. SSE（Server-Sent Events）

见 ElderCare 讲稿决策 7。一句话：**LLM 流式 = SSE，复杂实时交互 = WebSocket**。

**本项目**前端用 `fetch + ReadableStream.getReader()` 手动解析，**不是浏览器原生 EventSource**——因为 EventSource 只支持 GET，不支持 multipart 上传图片。

### 5. async generator + asyncio.to_thread

- `async def f() -> AsyncIterator[T]`：可以 `async for x in f()` 消费
- `await asyncio.to_thread(sync_func, *args)`：把同步函数扔线程池，不阻塞 event loop
- **本项目**：planner 是 async gen，VLM/LLM/RAG 都用 to_thread 包装

### 6. Pydantic vs SQLModel

- Pydantic：纯数据验证 + JSON 序列化
- **SQLModel**：Pydantic + SQLAlchemy，**一个类同时是 API schema 和 ORM 模型**
- 本项目 4 表都用 SQLModel，避免重复定义

### 7. Tailwind v4

- v4 用 `@tailwindcss/vite` 插件 + `@import "tailwindcss"`，**无需 tailwind.config.js + postcss**
- 比 v3 简单 80%

### 8. JSDoc 渐进式类型

- 不上 TypeScript（学习曲线 + 编译配置成本）
- 用 `/** @typedef {Object} Pet @property {string} name */` 写类型注释
- VSCode 原生支持，hover 提示 + 参数校验
- 未来迁 TS 时类型注释直接转 `interface`

### 9. Chroma metadata filter

```python
collection.query(
    query_embeddings=[...],
    n_results=5,
    where={"species_cat": True, "emergency": True}  # 多字段用 $and 包裹
)
```

Chroma metadata 只支持 str/int/float/bool/None，**不支持 list**——所以把 `species: [猫,狗]` 转成 `species_cat: true, species_dog: true` 两个 bool 字段。

### 10. localStorage 持久化模式

```javascript
// per-pet session 映射
localStorage.setItem('petpal.sessions', JSON.stringify({1: 'uuid-1', 2: 'uuid-2'}))
// 当前 active pet
localStorage.setItem('petpal.activePetId', '3')
```

简单可靠，5-10MB 容量对宠物档案场景足够。

## 七、深度问答 40+

### A. 多模态 Agent 设计类

**Q1：你的 agent 和市面上的 chatbot 有什么区别？**

> 本质区别在**决策权**：chatbot 是固定流水线，LLM 只负责生成；agent 是 LLM 自主决策——看到用户输入后，**它决定**调哪些工具、按什么顺序、调几次。我的 planner 用 OpenAI 真 Function Calling 协议（不是 prompt 工程模拟），LLM 返回 `tool_calls` 列表，我执行后把结果 append 回 messages 让 LLM 继续——这是 ReAct 风格 agent 的标准实现。每一步 tool 调用都持久化到 `chat_sessions.tool_calls_json`，决策可审计。

**Q2：为什么不用 LangChain / AutoGen 这些框架？**

> 三个考虑：① **可控性**——框架封装太多，调 prompt 和 agent loop 时很难精细控制；② **学习成本**——直接用 OpenAI SDK 5 个核心 API 调用就能搭出 agent loop，框架反而多一层学习；③ **教学价值**——自己写一遍 ReAct loop 比"调 LangChain 黑盒"对面试官更值钱。我的 `run_agent_stream` 函数 200 行覆盖了完整 agent 主循环，可读性 + 可调试性都更高。生产场景如果要做企业级 agent（多 agent 协作、回溯、长上下文管理）才考虑 LangGraph / AutoGen。

**Q3：VLM 单图能识别情绪 / 体态 / 疼痛吗？局限是什么？**

> **能也不能**。能识别**静态肢体信号**——耳朵位置、瞳孔大小、尾巴姿态、姿势、毛发状态、面部表情。不能识别**动态/音频信号**——尾巴抽打频率、呼噜、低吼、呼吸节奏。所以我特意让 emotion task 输出 `candidate_emotions + confidence`（不断言）、加 `caveat: 单张图无法判断动态信号`——这是**诚实工程权衡**。FGS 量表能用 prompt 化的原因正是它**只依赖静态视觉特征**，不需要观察时序。

**Q4：5 个 task 路由准确率怎么保证？**

> 用 cheap gpt-4o-mini 文本调用做分类，给了清晰的 task 描述。**关键 trick**：多轮对话中 router 会读取最近一条 assistant content 做 hint 推断——比如上一轮 agent 说「请上传一张全身侧照」，下一轮用户只发图，router 会从 hint 里的"全身侧照"关键词推断 task = bcs，**继承上下文**。具体实现：维护 `_HINT_KEYWORDS` 字典（bcs / pain_fgs / emotion / symptom 各自关键词），匹配到就路由对应 task，没匹配到才 default chat。

**Q5：为什么 chat 任务也调 VLM 看图？**

> Chat 是闲聊但**用户分享照片就期望 agent 看到内容**——比如「我家猫可爱吗？」。如果 chat 不看图，agent 就只能泛泛回应"很可爱啊"，体验差。我让 chat task 走 symptom prompt（拿到 observation 字段），但 prompt 明确**不医学化**——agent 用 observation 做自然温和的回应（「这只橘猫眼神好有气场」）而不是给医学建议。**这是产品决策**：图永远不浪费，但分场景使用。

### B. Function Calling 类

**Q6：怎么避免 LLM 调错 tool 或调多余的 tool？**

> 三层保护：① **prompt 工程**——SOP 描述每个 task 的工具调用流程；② **schema 自带约束**——每个 tool 的 description 里写「只有 X 时才调」（如 `reanalyze_image` 的 description 明确「前提：本会话必须已有图片」）；③ **工程兜底**——`saved_event_keys` set 拦截重复 save_pet_event；`_save_pet_event` 检测 FGS=0 自动 skip；duplicate/skipped 的 result 前端自动隐藏。**LLM 不可靠时代码兜底**。

**Q7：agent loop 怎么避免死循环？**

> `max_iter=5` 硬限制。**5 轮是经验值**——绝大多数 case 在 2-3 轮内自然终止（LLM 调 1-2 个 tool 后给 final）。达到 5 轮还没终止 → yield `max_iter_reached` 事件 + 给主人「请简化问题」的提示。同时 `saved_event_keys` 去重防止 LLM 在同 tool 上重复调用消耗 iter。生产场景如果常达上限，要分析 prompt 是否有歧义。

**Q8：tool 执行失败怎么办？**

> 每个 tool impl 内部 try/except，失败返回 `{ok: False, error: "..."}` 而不是抛异常——这样 LLM 看到结果能自己决策（再试 / 给主人解释 / 走 fallback）。比如 `find_nearby_clinic` 高德 API 报错 → 返回 error，LLM 在 final 里说"地图查询失败，建议您自己搜索附近宠物医院"。**LLM 处理失败比 hard exception 优雅**。

**Q9：多个 tool 能并行调用吗？**

> OpenAI tools 协议**支持** LLM 在一轮返回多个 tool_calls，我代码也支持遍历执行。但实际情况下 LLM 通常一轮调 1-2 个（先 retrieve_vet_knowledge 拿信息，下一轮再 save_pet_event）。**并行调用收益有限**——除非有明显并行场景（同时查 RAG + 查地图），通常顺序调更稳。

**Q10：tool description 怎么写才让 LLM 调对？**

> 三个原则：① **明确"什么时候用"**——比如 `find_nearby_clinic` 写「仅当用户明确给出具体地址才调」；② **写明限制**——比如 `reanalyze_image` 写「本会话必须已有图片，否则报错」；③ **参数描述清晰**——比如 `save_pet_event.event_type` 列出所有合法值。我踩过坑：早期 `reanalyze_image` description 太宽泛，LLM 会在没图时也调，触发"no image"错误。

### C. FGS 临床量表类

**Q11：你怎么知道 FGS 这个量表的？**

> 设计 emotion / pain 任务时，用户反馈"猫紧张和愤怒分不清"。我查文献时找到 Evangelista 等 2019 年发表在 **Scientific Reports** 的论文 _Facial expressions of pain in cats: the development and validation of a Feline Grimace Scale_。这是**同行评议**的科学量表——5 个 action units × 0/1/2 评分，total ≥ 4/10 提示临床显著疼痛。我意识到这种「视觉特征 + 离散评分矩阵」**完美适合 VLM 输出 schema**——比让 LLM 凭印象判断"疼不疼"科学。

**Q12：FGS prompt 化具体怎么实现的？**

> 我把论文训练手册里 5 个 AU 的 0/1/2 判定标准（耳朵朝前 vs 略外侧 vs 完全侧贴 等）**直接抄进 prompt**，让 gpt-4o-mini 看图后对号入座。输出 schema 是 Pydantic 严格校验：
> ```python
> class PainFGSResult(BaseModel):
>     action_units: list[ActionUnit]  # 5 个 AU
>     total_score: int = Field(..., ge=0, le=10)
>     normalized: float
>     interpretation: str  # 解释 vs 0.39 阈值
>     source: str = 'Evangelista et al. 2019, Scientific Reports'
> ```
> Pydantic 解析失败自动重试 1 次（temperature=0 + 错误提示）。**这是把临床量表工程化的标准模式**。

**Q13：通用 VLM 评 FGS 准吗？怎么验证？**

> 准确率验证是 P7 评测要做的——目前测试 19 张猫照片观察输出合理（放松猫给 0 分、紧张猫看角度给 0-2 分）。**诚实瑕疵**：FGS 论文是专业兽医标注的，gpt-4o-mini 是通用模型，**单图准确率应该比专业标注低**。改进路径：① 给 VLM 看 PDF 里 5 张对照图当 few-shot（v2 优化）；② 用 Roboflow 标 500 张训自训分类器（V2）。**面试讲点**：「VLM baseline 准确率 X%，自训 EfficientNet 提升到 Y%」是典型 P7 评测故事。

**Q14：为什么不直接训分类器？**

> **学生项目工程量考虑**：自训 5 个 AU 分类器需要 ≥500 张标注图 × 5 个维度 = 2500 张高质量兽医标注，没有兽医协助做不到。用 prompt 工程让通用 VLM「对号入座」是 80% 效果 / 5% 成本的解。**面试讲点**：「我评估过三条路径（Teachable Machine / Roboflow / VLM prompt），最终选 VLM prompt——专业量表 + 通用模型，落地最快」。

### D. RAG 类

**Q15：你的兽医知识库怎么来的？**

> 完全**手写 130 条**——11 个主题文件，按系统分类（消化 / 呼吸 / 皮肤 / 眼耳 / 全身 / BCS / 疫苗 / 急救 / 日常护理 / 行为 / 品种）。每条 200-400 字（中文 RAG chunk 最佳长度），含 8 字段 YAML frontmatter：species（猫狗）、severity（low/medium/high）、age_group、emergency（bool）、tags、source（PetMD / Merck / Cornell 等）、source_url。**不爬国内站**——反爬 + 版权 + 伦理三重风险，作为简历项目不值得。

**Q16：YAML frontmatter 是什么？为什么用它？**

> 一种结构化数据描述方式，markdown 文件里用 `---` 包围 YAML 块：
> ```markdown
> ## 症状-呕吐-带血
> ---
> species: [猫, 狗]
> severity: high
> emergency: true
> ---
> ### 表现
> ...
> ```
> 优势：① **人类可读**（写知识库时不用学专门 DSL）；② **机器可解析**（PyYAML 直接 parse）；③ **未来扩字段不破坏旧条目**。我的 `ingest_kb.py` 用正则切 `^## ` 块 + yaml.safe_load 解析 frontmatter，130 条入库 5 秒。

**Q17：Chroma metadata 只支持基础类型，怎么存 list？**

> 转 bool 字段。`species: [猫, 狗]` → `species_cat: true, species_dog: true`；`age_group: [幼, 成, 老]` → `age_young/adult/senior: true`。这样 metadata filter `where: {species_cat: true, emergency: true}` 能高效命中。tags 字段用逗号拼接成 string，仅作展示用不参与 filter。

**Q18：知识库这么小（130 条），RAG 效果靠谱吗？**

> 130 条够 MVP——关键是**领域聚焦**而不是数量。我每条覆盖一个高频场景（呕吐 / 腹泻 / BCS X 分 / 疫苗 X 类），用户最常问的 80% 问题能命中。诚实评估：长尾问题（罕见品种特性、专业术语）会漏，需要 P1.5 翻译 PetMD 扩到 200-300 条。**面试讲『质量 > 数量』**——130 条同行评议来源的知识比 10000 条爬来的更可信。

### E. 工程兜底类（⭐ 高区分度）

**Q19：你说工程兜底，具体兜哪些？**

> 5 个关键兜底：
> 1. **「光说不做」检测**：LLM 输出过渡语但 tool_calls 空 → 注入 system 提示重试
> 2. **save_pet_event 去重**：同 `(pet_id, event_type)` 一轮调过就拒
> 3. **final 自动补存**：task 需存事件但 LLM 没调 → 后端 silent 补存（UI 不显示）
> 4. **FGS=0 自动 skip**：`_save_pet_event` 检测 total_score=0 拒绝入库（noise 事件）
> 5. **历史图自动当本轮图**：本轮无图 + session 有图 → planner 直接复用历史磁盘路径走 VLM
> 
> **核心思想**：LLM 行为非确定性，关键业务规则不能完全靠 prompt，要代码强制。

**Q20：你说 prompt 改了 8 版，最大的教训是什么？**

> **少堆死规则，多用工程兜底**。早期我把规则一条条加进 prompt（❌ 绝对禁止 X / 必须 Y），结果 LLM 输出越来越机械、模板化，用户反馈"没温度"。后来转向**引导式**——「你不是冷冰客服，是懂宠物的朋友」+ 几个 motivation 例子，**让 LLM 自由发挥共情**；关键业务规则（save_pet_event 必调、不存重复事件、FGS=0 skip）用代码兜底。最终 prompt 从 3300 字砍到 2000 字，效果反而更好。

**Q21：「光说不做」检测具体怎么做？**

> 文本特征匹配 + 长度判断：
> ```python
> _TRANSITION_HINTS = ('稍等', '让我', '我先', '我再', '我查', ...)
> def _looks_like_transition_only(text):
>     if len(text) > 200: return False
>     if not any(w in text for w in _TRANSITION_HINTS): return False
>     has_specifics = any(d in t for d in ['24 小时', '禁食', '## ', '- '])
>     return not has_specifics
> ```
> 触发后 yield thinking 事件 + 注入 system message 「请立即调对应 tool，只说不做等于放弃任务」，continue 进下一轮 iter。**这种 trigger word 匹配看起来粗糙但有效**——LLM 偷懒的句式重复度极高。

**Q22：自动补存为什么不发 SSE 事件？**

> **UI 视觉干净 > 完全透明**。如果补存事件也展示成 tool 卡片，用户会看到「LLM 没调 → 系统补 → 又一张卡片」的尴尬流程。我选择 **silent 补存**——db 里 note='[auto-saved by backend]'，可审计；UI 上彻底隐藏。**面试讲点**：「我用静默兜底保护用户体验，但保留 db 审计追溯能力」。

**Q23：`str.format` 那个 bug 怎么发现的？**

> 用户报「Agent 出错: 'symptom_desc'」。一开始我搜代码没找到访问 `symptom_desc` 的地方。我让用户在 uvicorn 控制台看 traceback，定位到 `SYSTEM_PROMPT_TEMPLATE.format()` 那行——Python `.format()` 把 prompt 里的 `{'symptom_desc': '...'}` 字典字面量当成 placeholder 解析，找不到 `symptom_desc` 这个 format 参数就 KeyError。**修复**：改用 `str.replace('{task}', task)` 三次，避开 format 语义。**这是 prompt 模板 + Python 内置方法天然冲突的典型坑**。

### F. SSE / 前端类

**Q24：为什么不用 EventSource 而用 fetch?**

> EventSource 只支持 GET 请求，**不支持 multipart 上传图片**。chat 接口需要 `pet_id + session_id + text + image`，必须 multipart POST。所以用 `fetch + ReadableStream.getReader()` 手动解析 SSE 帧：
> ```javascript
> const reader = res.body.getReader()
> const decoder = new TextDecoder()
> let buf = ''
> while (true) {
>   const { done, value } = await reader.read()
>   if (done) break
>   buf += decoder.decode(value, { stream: true })
>   const parts = buf.split('\n\n')
>   buf = parts.pop()
>   for (const part of parts) {
>     if (part.startsWith('data: ')) onEvent(JSON.parse(part.slice(6)))
>   }
> }
> ```
> 缓存上次未完整的事件帧，下次拼接——SSE 解析的标准做法。

**Q25：前端卡片流怎么设计的？**

> `UIBlock[]` 数组 + 多 kind：`user / task / vlm / thinking / tool / assistant`。SSE 事件来 → setBlocks 追加或更新对应 block。比如：
> - `task_classified` → 追加 `{kind: 'task'}`
> - `tool_call` → 追加 `{kind: 'tool', status: 'running'}`
> - `tool_result` → 找到上面那张卡 update `status: 'done', result: ...`
> - `final_answer` → 追加 `{kind: 'assistant'}`
> 
> **关键**：tool_call 用 `tool + iter` 作 key 关联后续 result，duplicate/skipped 的 result 直接从数组移除。

**Q26：刷新页面后历史怎么还原？**

> chat_sessions 表完整存了每个 message（role / content / tool_calls_json / image_url / task / is_intermediate / vlm_output_json）。刷新页面后调 `GET /api/sessions/{id}/messages` 拿全部 message，遍历重建 UIBlock 数组。**关键字段**：
> - `is_intermediate=true` → 还原为 thinking 卡（不是 final）
> - `vlm_output_json` → 还原 user message 缩略图的徽章（BCS X 偏胖）
> - `tool_calls_json` → 还原 tool 卡片（含完整 result，2KB 截断）

### G. 数据库 / 多 entity 类

**Q27：events 和 reminders 为什么分两个表？**

> **语义不同**：events 是「**已发生**」的不可变事件（BCS 评分历史 / 症状记录 / 疫苗接种）；reminders 是「**未发生**」的计划提醒（疫苗到期 / 驱虫日程）。两者用不同状态机：
> - events：只能 append / soft delete
> - reminders：可改 scheduled_at / 可消（notified=true）
> 
> 混在一起会让查询逻辑混乱（「查事件时间线」要排除 reminders）。**产品决策决定数据模型**。

**Q28：软删怎么实现？为什么不硬删？**

> Pet 表加 `deleted_at: datetime | None`，删除时 set 当前时间。List API 默认 `where(deleted_at.is_(None))` 过滤；`?include_deleted=true` 能看全部。**为什么软删**：pet_events 关联了大量历史事件（FK pet_id），硬删会丢失审计数据。生产场景误删时 `POST /api/pets/{id}/restore` 能恢复。

**Q29：头像上传踩过什么坑？**

> 三个坑：① **大图问题**——用户上传 27MB 原图，超过 VLM API 限制，Pillow resize 到 512×512；② **缓存问题**——更新头像后浏览器还显示旧图，文件名用时间戳 `avatar_{int(time.time())}.jpg` 防缓存；③ **Vite proxy**——`/static/` 路径前端访问需要在 vite.config.js 加 proxy 转发到 8000，**我漏配过这条**，导致头像 404 显示 alt 文本。

**Q30：chat_sessions schema 演进？**

> 初版只有 4 字段（id / pet_id / role / content / tool_calls_json）。P5 加了 5 个字段：
> 1. `session_id: str(36)` — UUID 把多条 message 分组成一次"对话"
> 2. `image_url` — 用户上传图的 /static URL
> 3. `task` — 路由分类结果
> 4. `is_intermediate: bool` — 区分中间 thinking vs final assistant
> 5. `vlm_output_json: str` — user msg 附带 VLM 输出（刷新还原徽章）
> 
> 用 sqlite3 ALTER TABLE 平滑迁移老库不丢数据。

### H. Prompt 工程类

**Q31：你写过几版 prompt？最大教训？**

> 8 版。最大教训：**少堆死规则**。详见决策 11 / Q20。

**Q32：「每轮 1 次」这种中文措辞有什么问题？**

> 中文「每轮 / 每次」有歧义——LLM 可能理解为「整次会话 1 次」（导致第二轮新症状不存事件）或「每条消息 1 次」（我本意）。后来改成「**对你当前要回答的这条用户消息**」，指代精确零歧义。**面试讲点**：「prompt 是给 LLM 看的，中文模糊词必须避免」。

**Q33：你怎么决定 prompt 哪些规则要保留，哪些删掉？**

> 三个判断标准：① **频繁触发的具体 bug** → 保留（如「不要在 final 后说要做 X」）；② **风格 / 人设描述** → 保留（朋友式 / 共情发挥 / 用宠物名字）；③ **过度限制的条目** → 删（早期写「绝对禁止超过 30 字」让 LLM 不写中间 content）。**核心方法**：每次改 prompt 都问自己「这条规则换工程兜底能做吗」——能做就删 prompt 加代码。

**Q34：你说 prompt 从 3300 字砍到 2000 字效果反而更好，为什么？**

> 三个原因：① **LLM attention 集中**——prompt 短，模型对每条规则的 attention 权重更高；② **少冲突**——规则多了容易自相矛盾（早期我同时说「thinking 详细分析」和「thinking ≤ 30 字」），LLM 选择困难；③ **给空间发挥**——LLM 本能共情和组织能力强，规则越死越机械。**Less is more 在 prompt engineering 里特别真**。

### I. 系统架构 / 工程

**Q35：你的 OpenRouter 调用走什么模型？为什么不直接用 OpenAI？**

> **gpt-4o-mini via OpenRouter**。理由：① 国内访问 openai.com 不稳定，OpenRouter 是聚合代理，国内访问稳定；② 同时接 200+ 模型（gpt / claude / llama / qwen），需要切模型时改一个 base_url 就行；③ 计费透明，不需要先充 OpenAI credits 再申请 API。**讲面试**：「OpenRouter 是国内做 AI 应用的事实标配——节省 VPN / 切模型成本」。

**Q36：API key 怎么管理？**

> `.env` 本地存，`.gitignore` 排除掉。`python-dotenv` 启动时加载，全程 `os.getenv()` 读取。`.env.example` 进 git 作为模板（key 留空）。**永远不在对话里贴 key**——用户的安全偏好。生产部署到 HF Spaces 用平台 Secrets 注入。

**Q37：怎么调试 SSE 流式接口？**

> 三种方式：① **TestClient stream**——FastAPI 的 `client.stream('POST', ...)` 内存里模拟 SSE，scripts/test_stream.py 跑了 3 个 case 验证 12+ 种事件类型；② **直接 yield 拿 async generator**——`async for ev in run_agent_stream(...)` 不经 HTTP 层，最贴近后端逻辑；③ **浏览器 Network tab** 看 EventSource Stream 帧。我三个都用过，分别覆盖不同测试层级。

**Q38：怎么测试整个系统？**

> 目前做了 4 个 smoke test：
> - `test_db_api.py`：DB + 4 表 CRUD（含软删 / 头像上传 / 事件时间线）
> - `test_retriever.py`：三阶段 RAG（5 个 query × no-rerank + with-rerank 对比）
> - `test_vlm.py`：VLM 4 task 输出 schema 校验（7 个 case）
> - `test_agent.py`：agent 端到端（5 个 case，含 chat / symptom / emotion / bcs / emergency）
> - `test_stream.py`：SSE 事件流 + sessions API
> 
> **未做但应该做**：pytest 单元测试、Playwright E2E。这是工程化下一步。

**Q39：你的 token 成本估算？**

> 单次完整 chat（含 VLM + RAG + 2-3 个 tool + final）约 ~$0.005-0.01：
> - VLM 1024×1024 图：~$0.001
> - 路由 cheap text：~$0.0001
> - 主 LLM agent loop（2-3 轮）：~$0.003-0.008
> - RAG 检索：本地计算无 token 成本
> 
> 130 条知识库 ingest 一次性 ~$0.05（embedding 用本地 BGE 模型，零 token 成本）。

### J. 业务 / 产品类

**Q40：怎么保护用户隐私？**

> 三层措施：① **本地优先**——BGE embedding + Chroma 都本地跑，文本不出本地；② **API 最小化**——VLM 只发图 + 必要 prompt，不发用户身份；③ **db 隔离**——用户上传的图存到 `data/uploads/`（gitignore），不会进代码仓库。**未做**：用户能主动删除会话历史。

**Q41：为什么砍掉 send_alert_email？**

> 原本设计「急诊场景给主人发邮件」，但发现**主人就在 chat 前面看**——再发邮件给他自己很怪。**正确用法**：邮件应该用在「**日程提醒**」（疫苗到期 / 驱虫提醒）—— P6 会重新设计成 `schedule_reminder` tool 接 APScheduler。**面试讲点**：「我做了产品决策的减法——P5 砍掉急诊邮件，把 SMTP 工具留到 P6 做提醒，工具复用价值最大化」。

**Q42：项目最难的地方是什么？**

> **prompt 工程**。技术上 agent loop 200 行就能搭起，VLM 调用 + RAG 都是套 API，最难的是**让 LLM 行为稳定**——同一个 prompt 在 v1-v8 我反复改，每次都有边际收益但每次也都引入新问题。最大教训是「LLM 不是程序员，prompt 不是代码」——堆规则只会让模型机械，应该给**人设 + 例子 + 工程兜底**。**这种迭代经验比技术堆叠值钱**。

**Q43：如果重做这个项目你会改什么？**

> 三点：① **更早做评测集**——P5 阶段我反复调 prompt 但没有量化对照，全凭主观感受。如果早做 LLM-as-judge 评测（同 ElderCare 那套），改 prompt 时能看 task / tool / final 准确率数字；② **更早做工程兜底**——前 5 个 prompt 版本太信任 LLM 听话，到 v6-v8 才补足兜底，重做的话直接 v3 就上工程兜底；③ **避免 chat_sessions schema 后期 migration**——P3 设计时多想几步加上 is_intermediate / vlm_output_json，P5 就不用 ALTER TABLE。

### K. 个人成长

**Q44：你从这个项目学到了什么？**

> 三件事：
> 1. **「真 agent」和「chatbot」的差距远比想象大**——LLM 自主决策一引入，整个系统就要为非确定性兜底，prompt / 工程 / UI 都要重新设计
> 2. **多模态应用的工程化**——VLM 不只是 API 调用，prompt schema 设计、图片预处理、错误重试、成本控制都是细节
> 3. **「诚实工程权衡」比「全线碾压」可信**——我承认 emotion 单图局限、承认 FGS 通用模型准确率不如专业标注、承认 prompt v1-v7 都失败过——这种坦诚比堆参数更打动面试官

**Q45：你怎么决定项目什么时候算完？**

> **MVP 跑通 + 简历能讲清楚故事**就算完，不追求"全做完"。P0-P5 完成了「拍照 + 文字 → agent → 答复 + 入库」核心闭环，已经能演示 + 讲故事。P6/P7（提醒 + 评测 + 部署）是优化项不是必需。**面试讲点**：「我倾向做完 80% 核心 + 留 20% 优化空间**让面试官提问**——这比 100% 做完没东西可聊更好」。

## 十、答辩金句

### 开场（被问"介绍下你的项目"）

> "我做了一个**真正的多模态 AI Agent**——PetPal 智能宠物管家。和市面上的 chatbot 不同，它是**LLM 自主决策**调哪些工具、按什么顺序：看到主人拍照 + 文字后，先用 cheap 模型分类到 5 个 task（症状 / 体态 / 疼痛 / 情绪 / 闲聊），再调 VLM 看图，然后主 LLM 自主编排 6 个 tool 完成端到端服务——查兽医知识库 / 调高德地图 / 写时间线 / 重看图 / 排提醒。
>
> **项目最强差异化**：临床疼痛量表 FGS（Evangelista 2019 同行评议论文）**编码为 VLM 输出 schema**，让通用 gpt-4o-mini 输出医学级评估——5 action units × 0/1/2 + total/10 + 临床阈值。这是'**科学量表 prompt 化**'的典型案例。
>
> 还做了 P6 时序对比图（多次拍照看 BCS / 体重 / FGS 变化，**多 entity 长期跟踪**），P6.2 APScheduler 提醒系统（疫苗 / 驱虫 / 体检），P1.5 把知识库从 130 翻倍到 **257 条**（128 条 MSD Veterinary Manual 翻译，Merck Manual 是兽医百科金标准）。
>
> **P7 评测拿到硬指标**：retriever 32 query × 4 配置对比，hybrid+rerank hit@5 96.7%，rerank 单独贡献 +27% hit@1；agent E2E 10 case × 3 runs **6 轮迭代到 96.7% pass**——每一轮都暴露具体工程问题，修完重跑数据可量化。
>
> 技术栈：Python + FastAPI + SQLModel + ChromaDB + 本地 BGE embedding + OpenRouter + APScheduler，前端 React 19 + Vite + Tailwind v4 + Recharts，全异步流式 SSE。"

### 进阶讲点（被问"工程上最难的部分"）

> "**LLM 的非确定性 + 工程兜底**——这不是用了什么新技术，但是 LLM 应用工程化的精髓。
>
> 我做了 7+ 个工程兜底：光说不做检测 + system 注入 retry / saved_event_keys 同 type 去重 / 体重 note 兜底正则提取 / 历史 VLM 进 context（避免重跑 VLM）/ reanalyze 智能拒绝（同 task + 空 focus）/ ingest 重复 title 自动 dedup / dry-run 邮件 db 留痕。
>
> 这些兜底**不是 prompt 改不到位的修补**，而是**主动设计——接受 LLM 50-70% 命中率的现实**。我 P7 评测主动暴露 a02 / a03 这种'明确事件型 query 不查 RAG'是 prompt 引导无法触达的 corner case，主动选择**不堆死规则**避免输出模板化（P5 已踩坑）——这是工程权衡，不是搪塞。"

### 进阶讲点（被问"评测怎么做的"）

> "我做了两层评测——retriever 数据驱动 + agent E2E 人工口径校准。
>
> Retriever 32 query × 4 配置对比，发现 rerank 单独贡献 +27% hit@1，hybrid+rerank 整体 hit@5 96.7%。还做了 OOK robustness——用 '我家鹦鹉打喷嚏' 这种非猫狗 query，验证 top-1 score 显著低于 in-KB（0.347 vs 0.979 差距 0.6+），可作 agent 判断'无相关知识'的工程信号。
>
> Agent E2E 评测**故事最有意思**——10 case × 3 runs，pass rate 从 **40%** 单次跑 → 53% 量化非确定性 → 80% 评测口径校准（task 放宽 + final any-of）→ 83% → 80%（prompt 实验）→ **96.7%（修真 bug）**。**6 轮迭代每轮都有可讲的工程决策**：从'发现 sync 缺 stream 同款兜底'到'历史 VLM 进 context 让 LLM 看上次分析无需 reanalyze'，闭环清晰。"

### 遇到不会的问题

> "这个我没深入实践过，但我的理解是 [说出能想到的方向]。如果项目要做到那一层，我会从 [具体路径] 入手研究。"
>
> 永远**不要说"不知道"然后停住**，要给出"我会怎么调研"的答案。

### 收尾（被问"还有什么要补充"）

> "我想强调三点：
> 1. **真 agent 和 chatbot 的差距远比看起来大**——LLM 自主决策一引入，整个系统就要为非确定性兜底，prompt / 工程 / UI 都要重新设计
> 2. **prompt 工程是个『减法过程』** —— 我改了 8 版 prompt，从 3300 字砍到 2000 字，**效果反而更好**——堆规则只会让 LLM 机械
> 3. **诚实工程权衡比全线碾压可信** —— 我承认 emotion 单图局限、承认 FGS 通用模型不如专业标注、承认 8 版 prompt 都有失败案例——**这种坦诚比堆参数更打动我自己**，希望也能打动您

**最后送一句话**：

> 面试讲项目，**讲'为什么这么选'比讲'怎么实现'重要 10 倍**。
>
> PetPal 最值得讲的不是「我用了 VLM 和 Function Calling」（任何人调 API 都能做），而是：
> 1. **诚实承认 LLM 不可靠 + 设计工程兜底**
> 2. **把临床科学量表 prompt 化**
> 3. **Prompt 工程的减法迭代经验**
> 4. **Context Engineering > Tool Engineering 的判断**
>
> 这些**判断力**才是 AI 应用岗的核心能力。
